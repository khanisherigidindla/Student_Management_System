1. What We Are Building ? 

We are building a Student Management System.

From the client’s perspective:

“I want a web page where I can add students, see them in a table, edit them, and delete them. The data should be stored permanently.”

That means:

The browser is only a UI.

The server handles logic.

The database stores the data permanently.

This is a classic 3-tier architecture:

Frontend (Browser)
        ↓ HTTP
Backend (Flask API)
        ↓ SQL
Database (MySQL)

Each layer has a single responsibility.

2. Clarifying Every Technical Term

Let’s define every major concept clearly.

Backend

Backend means:

A server program running on your machine

Written in Python

Handles incoming HTTP requests

Talks to database

Returns JSON

We will use:

Python

Flask (a micro web framework)

Flask

Flask is a lightweight Python web framework.

It allows us to:

Create routes (URLs)

Accept HTTP requests

Send JSON responses

Flask does NOT:

Automatically create database tables

Automatically validate inputs

Automatically structure project

We must handle those manually.

REST API

REST means:

We use HTTP methods properly

Each URL represents a resource

In this project:

Method	URL	Meaning
GET	/students	Get all students
GET	/students/1	Get student with ID 1
POST	/students	Create new student
PUT	/students/1	Update student 1
DELETE	/students/1	Delete student 1

Each endpoint corresponds to a SQL operation.

Database

We are using SQL database.

You can choose:

SQLite (file-based, simplest)

MySQL (server-based)

For a fresher project:
Use SQLite.

Why?

No installation overhead

Creates a .db file

Good for development

CORS

CORS = Cross-Origin Resource Sharing.

Problem:

Frontend runs on port 5500

Backend runs on port 5000

Browser blocks request for security

Solution:
Use:

from flask_cors import CORS
CORS(app)

This enables cross-origin requests.

4. Backend Deep Explanation (Pin to Pin)
File: backend/app.py

This file does 5 major things:

Create Flask app

Connect to database

Create table (if not exists)

Define API routes

Handle errors

Step 1: Initialize Flask
from flask import Flask, request, jsonify
app = Flask(__name__)

app is your server object.

Step 2: Connect to Database

If using SQLite:

import sqlite3

def get_db_connection():
    conn = sqlite3.connect("database.db")
    conn.row_factory = sqlite3.Row
    return conn

Explanation:

Connects to file database

row_factory allows dictionary-style access

Step 3: Create Table

You must ensure table exists.

CREATE TABLE IF NOT EXISTS students (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    email TEXT NOT NULL,
    course TEXT NOT NULL,
    marks INTEGER NOT NULL
);

Important:

id auto-generates

NOT NULL prevents empty insertion

Step 4: Implement CRUD Routes
GET /students

Flow:

Connect DB

Execute SELECT

Fetch all rows

Convert to JSON

Return 200

GET /students/<id>

Flow:

Extract id from URL

Run SELECT WHERE id=?

If no result → 404

Else return student

POST /students

Flow:

Read JSON body

Validate:

All fields exist

marks is integer

Insert into DB

Return 201 Created

PUT /students/<id>

Flow:

Check if student exists

Validate input

Run UPDATE

Return success

DELETE /students/<id>

Flow:

Check existence

Run DELETE

Return confirmation

8. Proper Error Handling

Backend must handle:

Case 1: Missing fields
→ 400 Bad Request

Case 2: Student not found
→ 404 Not Found

Case 3: Database error
→ 500 Internal Server Error

Return consistent JSON:

{
  "success": false,
  "message": "Student not found"
}

Consistency matters.

9. What Client Actually Cares About

Client doesn’t care about Flask.

Client cares about:

Does add work?

Does delete work?

Is data saved permanently?

Is it user-friendly?

Does it break?

You must think like:

“What if user deletes non-existing student?”

“What if user enters marks = -10?”

Final Advice From Senior to Junior

Do not rush to code immediately.

First:

Understand flow

Design endpoints

Write SQL queries manually

Test with Postman

Then connect frontend

If your API is weak, frontend will collapse.

If your validation is weak, system becomes unstable.

Build backend strong first.
